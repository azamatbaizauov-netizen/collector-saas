# ADR 0004: Канал задач менеджерам — общий Telegram-чат с видимостью собственника

**Дата:** 2026-06-16
**Статус:** Принято

---

## Контекст

По исходному CLAUDE.md рабочее место менеджера по дебиторке — **Виджет B в Битриксе** (дашборд «кому написать сегодня»), а Telegram — канал **только для собственника** (утренняя сводка).

На пилоте это не работает:

1. **В Битриксе нет данных по долгам.** Дебиторка живёт в Google Sheets (см. ADR 0003), обмена 1С↔Битрикс нет. Строить виджет поверх отсутствующих данных рано.
2. **Менеджеры не живут в Битриксе.** Они работают с телефона: WhatsApp с клиентами + Telegram между собой. Дашборд в CRM, куда они не заходят, не приживётся.
3. **Главная цель механики — давление присутствием собственника.** Задача не «дать менеджеру список», а сделать так, чтобы менеджер *понимал: собственник прямо сейчас видит, что бот ему напомнил*. Виджет в личной карточке Битрикса этого не даёт — он приватный.

Режим работы остаётся прежним (Мониторинг из CLAUDE.md): **бот напоминает менеджерам, должникам пишут сами менеджеры**. Система никому не пишет за менеджера.

---

## Решение

Канал задач менеджерам на пилоте — **один общий Telegram-чат** [собственник + бот + все МОПы]. Виджет B в Битриксе из плана пилота убираем (переносим в Фазу 2, когда в Битриксе появятся данные).

Механика прозрачности собственника — комбинация **1 + 2 + 4**:

**1. Общий чат, обращение по @username.**
Бот постит план дня в группу, адресно: `@adilbek сегодня напомни: Назаров ($5256, 32 дн), Дуйсебаев ($14217, 11 дн)`. Собственник — видимый участник группы, поэтому всё, что пишет бот, по определению на глазах у собственника.

**2. Ежедневная сводка собственнику в личку — только его клиенты, без подтверждения.**
Собственник Медет **ведёт своих клиентов сам** (строки «Медет» в Sheet) и хочет видеть их каждый день — шлём ежедневный дайджест **по его собственным должникам** в личный Telegram (`ownerTelegramUserId`, `MORNING_DIGEST`).

Что НЕ кидаем в личку: **разбивку по менеджерам** (кто сколько закрыл) — её Медет видит в общем чате (механики 1 и 4), дублировать не нужно.

Это информирование, а не запрос: **кнопку-подтверждение не используем** (иначе замьютит). Бот в общий чат постит факт `✅ сводка сформирована` сам.

Плюс личка по исключениям вне расписания: менеджер закрыл 0 из N, должник ушёл в STOP, систематический игнор плана.

**4. Публичный счётчик дня на cutoff.**
В заданное время (`OrganizationSettings`, напр. 12:12) бот постит в общий чат сводку покрытия:
`Адилбек 8/10 ✅ · Амир 5/12 ⚠️ · Медет 3/9 ❌`.
Собственник видит рейтинг, менеджеры видят, что их сравнивают публично. Это и есть целевое социальное давление.

Пункт **3** (явная шапка `👁 видит: <собственник>` в каждом сообщении) — опционально на первые недели для выработки привычки, потом убрать. Не часть постоянной механики.

## Поток (с учётом ADR 0003)

```
Google Sheet (polling по version) → нормализация
   ▼ воркер "план на сегодня" (утро)
Telegram-чат: адресные задачи по @username (механика 1)
   + ежедневная сводка собственнику в личку — его клиенты (механика 2, MORNING_DIGEST, без подтверждения)
МОПы пишут должникам в WhatsApp (свои рабочие номера)
   ▼ Green API webhook ловит исходящие
воркер "проверка покрытия" (cutoff, напр. 12:12)
   ▼
Telegram-чат: публичный счётчик N/M по каждому МОПу (механика 4)
```

---

## Линчпин: таблица соответствий идентичностей

Без неё ничего не связывается. На каждого менеджера связываем три идентичности (хранить в нашей БД, с `organization_id`):

| Имя МОПа в Sheet (free-text) | Telegram user_id | WhatsApp instance (Green API) |
|---|---|---|

- Имя из таблицы — свободный текст («Амир», «Амир Медет», «Медет») → сматчить с реальными людьми один раз вручную.
- Telegram user_id — узнаём при первом `/start` менеджера боту.
- WhatsApp instance — из `WhatsAppChannel`.

## Открытые вопросы (решить до реализации)

1. **Критерий «менеджер связался».** Старт — *любое исходящее WhatsApp на номер должника за окно [план … cutoff]*. Уточнение «именно про долг» через Haiku — вторым этапом. **Проблема звонков:** WhatsApp-звонки/видео мы не видим (e2e) → менеджер, который позвонил, ложно попадёт в «не связался». Нужна команда/кнопка «записать итог звонка», иначе несправедливые ❌ в публичном счётчике. Финальный критерий не зафиксирован.
2. **Детальный публичный callout по именам** («не связались: Адилбек → Назаров») — пока заменён агрегатным счётчиком (механика 4). Нужно ли называть конкретные пары публично или это слишком жёстко — решить с собственником на пилоте.

## Владение и настройка Telegram-бота

Бот — **наш продукт, один на всех клиентов**, создаём и держим мы (не клиент). Токен живёт в нашем backend (`.env`), клиенту не передаётся.

- **Создавать под общим/продуктовым Telegram-аккаунтом**, не под личным одного из партнёров (иначе бот привязан к человеку — bus-factor; перекликается с партнёрским соглашением из юр-чек-листа).
- Организация определяется по **`chat_id` группы** (`OrganizationSettings.telegramGroupChatId`). Пилот — один чат; Фаза 2 — много чатов, один бот, роутинг по `chat_id`. Соответствует принципу «Локальное → Размещаемое без переписывания».
- **Privacy mode** в @BotFather (`/setprivacy` → Disable) или бот = админ группы — чтобы видеть все сообщения/реакции, а не только команды.

Действия клиента — минимальны: (1) добавить бота в групповой чат; (2) собственник один раз жмёт `/start` боту в личке, чтобы мы узнали его `telegramUserId` для исключений-алертов и еженедельного дайджеста (механика 2).

## Организационное условие (блокирующее)

Механика работает, только если менеджеры **официально предупреждены**, что рабочая переписка с корпоративного номера мониторится, а сводки идут собственнику. Это уже в юридическом чек-листе CLAUDE.md. Иначе «за мной следят» = конфликт.

## Отклонённые альтернативы

- **Виджет B в Битриксе (текущий CLAUDE.md).** Приватный, требует данных в Битриксе (которых нет) и захода менеджеров в CRM (которого не происходит). Не даёт эффекта присутствия собственника. Переносим в Фазу 2.
- **Личные сообщения боту каждому МОПу.** Приватно — теряется весь смысл «собственник видит». Подходит максимум для мягких напоминаний, не для механики давления.
- **Явная шапка `👁 видит` в каждом сообщении как постоянная мера.** Со временем замыливается; оставлена опцией на первые недели.
