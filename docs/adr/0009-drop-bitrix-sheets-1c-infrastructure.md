# ADR 0009: Отказ от Битрикса — инфраструктура на Google Sheets + 1С

**Дата:** 2026-06-29
**Статус:** Принято (требует подтверждения вторым партнёром перед мерджем кода)

---

## Контекст

Исходная архитектура (CLAUDE.md до этого ADR) строилась вокруг Битрикса: приложение для Битрикс24, Битрикс — source of truth, рабочее место финменеджера, маркетплейс Битрикс24 — канал дистрибуции Фазы 2.

Уточнение целевой аудитории это опровергает. **ЦА — оптовики на базарах.** У них нет Битрикса и не будет: их реальная база — **Google-таблица + 1С**. Строить надстройку над Битриксом — значит решать задачу для клиента, которого у нас нет.

Пилот это уже подтверждает на практике — он де-факто работает **без Битрикса**:

- Источник долга — Google-лист + 1С, а не Битрикс (ADR 0003). Обмена 1С↔Битрикс у клиента нет.
- Рабочее место менеджеров — общий Telegram-чат (ADR 0004) + WhatsApp; виджеты Битрикса (B) отложены и не используются.
- Запись `Reminder`/`Promise` в Битрикс висит в TODO (`apps/worker/jobs/inbound-message`), заблокирована на резолве `contactId` Битрикса — то есть Битрикс реально **тормозит**, а не помогает.

Битрикс в текущей архитектуре — мёртвый груз и источник блокеров.

---

## Решение

**Убираем Битрикс из продукта целиком. Инфраструктура строится на Google Sheets (+ опционально 1С).**

1. **Source of truth — таблица клиента (Google Sheets), 1С — опциональный источник** через Excel-выгрузку (как у пилота). 1С есть не у всех базарщиков; нижний общий знаменатель — Google Sheets. Наша БД, как и раньше, хранит только вычисляемое (рейтинги, касания, обещания, конфиг) — принцип «не дублируем источник» сохраняется, просто источник теперь лист, а не Битрикс.
2. **Рабочее место — WhatsApp + Telegram.** Менеджеры ↔ должники в WhatsApp; план дня, сводки, счётчик покрытия, подтверждения собственника — в Telegram (ADR 0004). Свой UI — максимум тонкая страница настроек/онбординга. **Веб-дашборд не делаем.**
3. **Дистрибуция Фазы 2 — standalone SaaS, прямые продажи** базарным оптовикам. Маркетплейс Битрикс24 как канал отпадает (ЦА там нет).
4. **Онбординг клиента (Фаза 2)** = доступ к их таблице (сервисный аккаунт, как в ADR 0003) + подключение WhatsApp-номеров (Green API QR) + бот в их Telegram-чат. Никакого OAuth-флоу Битрикса и сертификации маркетплейса.
5. **Формат таблицы клиента — решаем по месту.** Где-то навяжем свой шаблон при онбординге, где-то подстроимся под их формат через маппинг колонок. Жёсткого единого правила нет; нормализатор/маппинг (ADR 0003) становится центральным узлом multi-tenant.

## Последствия

- **`packages/bitrix` и связанный код выводятся из плана.** На пилоте он в основном TODO/неактивен. Что физически удалять или архивировать — отдельным шагом, не в этом ADR (это документ-решение, не рефакторинг).
- **CLAUDE.md переписывается** в несущих местах: главное архитектурное решение, Фаза 1/2, «Два продукта», текущая стадия, принципы №1 (source of truth) и №7 (Локальное→Размещаемое приложение Битрикс → один клиент→multi-tenant SaaS), стек, структура репо, модель данных, раздел «Использование Bitrix24 API», анти-паттерны, юр-чеклист.
- **Убираем из scope:** виджеты A/B, `placement.bind`, кастом-поля `UF_AICOL_*`, timeline-комментарии Битрикса, OAuth Битрикса, чек-лист маркетплейса.
- **Адаптеры остаются принципом.** `MessagingAdapter`/источник дебиторки за интерфейсом — по-прежнему да (ADR 0003). Просто `CrmAdapter`/`BitrixAdapter` больше не центральная абстракция.
- **Риск масштабирования смещается на таблицы.** Разные форматы листов у разных клиентов — главный вызов multi-tenant. Грязные данные (ADR 0003) теперь не исключение пилота, а норма, которую закладываем в онбординг.
- **Юр-чеклист:** пункты про IP/партнёрство/SaaS остаются; пункт про маркетплейс Битрикс снимается.

---

## Отклонённые альтернативы

- **Оставить Битрикс опциональным коннектором «на будущее».** Нарушает принцип «не обобщаем под несуществующего клиента». Для базарной ЦА Битрикс не нужен, а мёртвый код и абстракция `CrmAdapter` держат сложность без пользы. Если когда-нибудь появится клиент на Битриксе — добавим адаптер тогда (как и задумано для внешних систем).
- **Строить свой полноценный веб-дашборд вместо рабочего места.** Базарщики живут в WhatsApp/Telegram; отдельный веб-UI они открывать не будут. Дашборд — это работа без спроса.
