# ADR 0003: Источник дебиторки — Google Sheets через polling по версии

**Дата:** 2026-06-16
**Статус:** Принято

---

## Контекст

У пилотного клиента дебиторка ведётся не в Битриксе. Долги живут в коробочной 1С, а оперативный учёт менеджеры ведут вручную в Google-таблице «Дебиторка LR» (лист «Отчёт МОПа»). Обмена 1С↔Битрикс у клиента нет, в Битриксе данных по долгам пока нет вообще. Значит для MVP Коллектора единственный реальный источник дебиторки — эта таблица.

Клиент выдал нам доступ на чтение через сервисный аккаунт `sheets-reader@saas-collector-499415.iam.gserviceaccount.com` (роль Viewer).

Ключевые факты, которые определили решение:

1. **Файл — загруженный `.xlsx`, а не нативная Google Таблица.** Sheets API его не читает (`This operation is not supported... must not be an Office file`). Читается только через Drive API (`files.get_media`), затем парсится `openpyxl`.
2. **Менеджеры правят один и тот же файл онлайн.** Метаданные Drive показывают `version: 4281` и регулярные `modifiedTime` — это инкрементальные правки в существующем файле, а не перезаливки новых. Значит `fileId` и ссылка стабильны.
3. **Данные «грязные»:** в колонке «Сумма долга» смешаны валюты ($ и ₸), часть строк съехала на колонку, имя клиента — свободный текст с городом внутри, телефон в плавающем формате, поле «последняя оплата» — свободный текст.

Вопрос ADR: как видеть изменения в этом источнике, не заставляя людей что-то перезаливать.

---

## Решение

**Periodic polling по `version`/`modifiedTime` через Drive API. Никаких перезаливок руками.**

Воркер (BullMQ, по расписанию из `OrganizationSettings`) делает дешёвый запрос `files.get(fileId, fields="version,modifiedTime")`. Если `version` изменился с прошлого раза:

1. Скачиваем `.xlsx` через Drive API (`files.get_media`), парсим `openpyxl`.
2. Нормализуем строки (валюта отдельно от суммы, телефон → E.164, отсев съехавших строк).
3. Сравниваем с прошлым снимком по ключу (телефон) — находим новые/изменённые строки.
4. Изменения пишем в нашу БД и в timeline соответствующего контакта Битрикса.

**Что храним у себя:** последний `version` файла и снимок прошлого нормализованного состояния (хэш строки по ключу) — для diff'а. Это не нарушает принцип «не дублируем Битрикс»: дебиторка по определению живёт вне Битрикса (1С + ручной Sheets), а снимок — технический буфер для обнаружения изменений, а не вторая копия бизнес-данных.

**Доступ:** через Drive API нужны включённые в проекте `saas-collector-499415` Google Sheets API и **Google Drive API** (оба включены 2026-06-16). Scope сервисного аккаунта — `drive.readonly`. JSON-ключ — секрет, хранится вне репозитория, путь — в `.env` (`GOOGLE_APPLICATION_CREDENTIALS`), в git не коммитится.

---

## Что меняется / появляется в коде

1. **Адаптер источника.** Чтение таблицы инкапсулируем за интерфейсом (аналогично `MessagingAdapter`/`CrmAdapter`) — `GoogleSheetDebtSource` в отдельном пакете. Бизнес-код не ходит в Drive напрямую.
2. **`.env` / `.env.example`.** Добавить `GOOGLE_APPLICATION_CREDENTIALS`, `DEBT_SHEET_FILE_ID`.
3. **Воркер polling'а.** Расписание в `ScheduleJob`/`OrganizationSettings`; хранит последний `version`; при изменении — pull + нормализация + diff.
4. **Нормализатор.** Чистые функции (в духе `packages/rules`): парс валюты из суммы/текста, телефон → E.164, детект съехавших строк (число в колонке «МОП»), матчинг с контактом Битрикса по телефону (имя — ненадёжно).
5. **Снимок для diff.** Таблица с `organization_id`, ключом-телефоном и хэшем нормализованной строки + `last_seen_version`.

## Нормализация данных

Таблица грязная (см. контекст). Нормализатор — чистые функции (`packages/rules`), на вход строка Sheet, на выход либо нормализованная запись, либо строка в «нераспределённые» + `AuditLog`. Правила зафиксированы по анализу реальных данных (лист «Отчёт МОПа», 14 колонок, на 2026-06-16).

**Колонки таблицы:**

| Кол | Поле | Кол | Поле |
|---|---|---|---|
| A | Клиент (free-text + город) | H | Дата последней оплаты |
| B | МОП | I | Дней без оплаты |
| C | Город | J | Лимит в долг |
| D | Телефон | K | Превышение лимита (= E − J) |
| E | Сумма долга | L | Косяк (статус-флаг, обычно «OK») |
| F | Дата возникновения | M | Комментарии (free-text) |
| G | Обещанная дата оплаты | N | Сумма последней оплаты (free-text: «1072$ оплатил 09.05») |

**Валюта.** Пилот ведётся в **USD** — это дефолт (`CreditLimit.currency`/`Promise.currency` = `USD`). 99% строк — доллары: лимиты (J) круглые ($1000/2000/5000/10000), превышение `K = E − J` сходится во всех строках. Реальный потолок долларового долга (E) ≈ **$53 500**, основная масса ≤ $30 000.
- **Тенге-выбросы** (~6 строк, крупные клиенты владельца: 1,5–23 млн) **не конвертируем автоматически.** Помечаем `currencySuspect = true` и ведём вручную.
- Порог подозрения: **долг (E) > 100 000 ИЛИ лимит (J) > 50 000** → `currencySuspect`. Между настоящим USD-потолком ($53,5K) и тенге-выбросами (от 1,56 млн) чистый разрыв, порог 100 000 не задевает ни одного валидного долларового долга.
- Суммы храним как `BigInt` в минорных единицах (центы) + код валюты. Никакого `float`, валюты не складываем между собой.

**Телефон → E.164** (колонка D, ключ матчинга с Битриксом):
- Часто хранится как `float` (`77757349292.0`) → парсим `int`, оставляем только цифры.
- 11 цифр, начинается с `7` → `+7XXXXXXXXXX` (Казахстан).
- 12 цифр, начинается с `996` → `+996XXXXXXXXX` (Кыргызстан, есть в данных).
- Иначе (битый/10-значный/пустой — ~12 строк) → строка в «нераспределённые» + `AuditLog`, не теряем.

**Даты** (F — возникновение долга, G — обещанная оплата, H — последняя оплата): парсим напрямую как даты, пустая → `null`. G (обещанная дата) → кандидат на `Promise.promisedDateFrom/To` (ADR 0002).

**Free-text** (M «Комментарии», N «Сумма последней оплаты» — там и сумма, и дата текстом): храним **сырыми**, на импорте не парсим. Извлечение смысла (обещания, суммы, даты) — отдельным этапом через Haiku, не в нормализаторе.

**МОП** (колонка **B**) → `ManagerSheetAlias.normalizedAlias` (см. ADR 0005). Съехавшая строка (число/мусор в колонке МОП) → флаг строки + `AuditLog action: 'sheet.unmatched_manager'`, без молчаливого импорта.

## Регламент клиенту (организационная часть, блокирующая)

Polling работает, **пока менеджеры правят ТОТ ЖЕ файл**. Если кто-то зальёт новый файл отдельной загрузкой — у него будет новый `fileId`, и мы «ослепнем». Зафиксировать с клиентом: *«правьте эту же таблицу, не создавайте новую»*. При смене файла нам нужен новый `fileId`.

## Отклонённые альтернативы

- **Разовый импорт + просьба «перезаливайте файл».** Ручной труд людей, легко забыть, нет надёжного триггера на изменение. Polling по `version` даёт то же самое автоматически и бесплатно.
- **Конвертировать в нативную Google Таблицу.** Чище технически (работает Sheets API без Drive, встроенная история версий, чтение по диапазонам), но меняет привычку клиента и одноразово ломает ссылку. Держим как опцию на случай, если упрёмся в ограничения `.xlsx`. На пилоте «ноль изменений в работе клиента» дороже технической чистоты.
- **Webhook/push-уведомления Drive (`files.watch`).** Реактивно, без polling'а, но требует публичный HTTPS-эндпоинт, обновляемые каналы (истекают), и сложнее в отладке. Для частоты «дебиторка раз в N часов» overkill — polling проще и достаточен.
