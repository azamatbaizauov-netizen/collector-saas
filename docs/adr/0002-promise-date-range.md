# ADR 0002: Обещания оплаты как диапазон дат

**Дата:** 2026-06-12
**Статус:** Принято

---

## Контекст

В пилоте мы парсим ответы клиентов на напоминания о долге через Claude Haiku (`packages/ai/src/conversations/parse-reply.ts`). Если клиент говорит «завтра закину» — получаем точную дату, кладём в `Promise.promisedDate`.

Реальные формулировки клиентов чаще размытые:

- «через 1–2 дня»
- «4–5 числа»
- «к концу недели»
- «во вторник–среду»

Сейчас в схеме одна колонка `promisedDate DateTime NOT NULL`. На фразу «через 1–2 дня» модель вынуждена выбрать одну из границ. Это даёт два класса ошибок:

1. **Ложно нарушенные обещания.** Клиент сказал «1–2 дня», система запомнила +1 день. Через сутки воркер уже считает обещание `BROKEN`, рейтинг клиента падает, менеджер дёргает его зря — а клиент в своём праве платить ещё день.
2. **Преждевременные напоминания.** Клиент сказал «4–5 числа», система запомнила 5-е. Менеджеру не подсвечивается, что начиная с 4-го уже можно мягко спросить.

Неопределённость в речи клиента — это **информация**, и терять её при записи в БД нельзя.

---

## Решение

В модели `Promise` храним **две даты вместо одной**:

- `promisedDateFrom DateTime` — нижняя граница обещанного окна. С этого момента менеджеру можно мягко напомнить, если оплаты нет.
- `promisedDateTo DateTime` — верхняя граница. После этой даты обещание помечается `BROKEN`, рейтинг пересчитывается, эскалация финменеджеру.

**Точная дата** («завтра», «в среду», «15-го») → `from == to`. Это инвариант, который в коде создания `Promise` явно учитываем.

**Диапазон** («1–2 дня», «4–5 числа», «в начале следующей недели») → `from < to`. LLM возвращает обе границы; пост-валидатор проверяет `from <= to` и `from >= today` (см. пункт «Что меняется» ниже).

**Открытая отсрочка** («на днях», «как смогу») → это **не обещание**, а `REQUEST_DELAY`. `Promise` не создаём вовсе, фиксируем `Reminder.intent = REQUEST_DELAY` и ставим задачу менеджеру через N дней (N в `OrganizationSettings`).

---

## Что меняется в коде

1. **Схема (`packages/db/prisma/schema.prisma`).** Заменить `promisedDate` на `promisedDateFrom` + `promisedDateTo`. Индекс `(organizationId, promisedDateTo)` — для воркера, который ищет просроченные обещания.
2. **Промпт (`packages/ai/prompts/parse-reply/v2.md`).** Запрашивать `{dateFrom, dateTo}` вместо одного `promisedDate`. В контекст передавать дату с днём недели и таймзоной организации (по умолчанию `Asia/Almaty`). Старый `v1.md` оставляем — версии не удаляем.
3. **`parseReply()`.** Новая выходная схема Zod: `dateFrom`/`dateTo` (обе обязательны, если intent `PROMISE_TO_PAY`). Пост-валидатор:
   - `dateFrom < today` → сдвиг на следующий аналогичный день (вероятнее всего клиент имел в виду следующий месяц/неделю), пометка `needsManagerConfirm`.
   - `dateFrom > dateTo` → меняем местами.
   - `dateTo - dateFrom > 14 дней` → подозрительный парс, intent понижается до `OTHER`.
4. **Воркер «нарушенных обещаний».** Сейчас сравнивает с `promisedDate`. Должен сравнивать с `promisedDateTo`.
5. **Виджет A.** В карточке контакта показываем «обещал 50 000 ₸ к 13–14.06». Если `from == to` — показываем одну дату.
6. **Snapshot-тесты `parse-reply`.** Минимум 10 фикстур с разными формулировками (точная, диапазон, открытая отсрочка, прошедшая дата, неоднозначная).

## Отклонённые альтернативы

- **Консервативно брать верхнюю границу одной колонкой.** Проще, но теряет момент начала окна → менеджер не знает, что уже можно мягко напомнить. Главное — не лечит первопричину: в БД остаётся ложь о точности.
- **Хранить исходный текст и парсить на лету.** Дорого (вызов LLM на каждое чтение), непредсказуемо (один и тот же текст может дать разный результат), и нельзя нормально индексировать для воркера «найти просроченные».
- **Структурированная JSON-колонка `{exact|range, ...}`.** Гибче, но индексировать диапазон по JSON — больно в Postgres, теряем простоту запросов воркера.
