# ADR 0011: Скриптовый multi-tenant онбординг (per-org лист + состав + номера)

**Дата:** 2026-07-06
**Статус:** Принято (требует подтверждения вторым партнёром перед мерджем кода)

---

## Контекст

Пилот работает, и мы готовим почву к Фазе 2 (продажа standalone SaaS базарным оптовикам, ADR 0009). Аудит показал: каркас multi-tenant в БД стоит правильно (`organization_id` во всех таблицах, вебхук резолвит орг по `instanceId`, Telegram-бот берёт `telegramGroupChatId` из `OrganizationSettings` per-org, планировщик создаёт задачи по каждой орг). Но «последние метры» онбординга прибиты к одному клиенту и не дают завести второго без правок кода:

1. **Лист дебиторки прибит к env.** `getDebtSource()` читал единственный `DEBT_SHEET_FILE_ID` из env для всех орг — второй клиент со своей таблицей некуда подключить.
2. **Seed прибит к пилоту.** `seed.mjs` смешивал generic-часть (организация + настройки + расписания) с захардкоженным составом менеджеров Медета и требовал `PILOT_ORGANIZATION_ID`.
3. **Скрипт номеров** (`onboard-channels.mjs`) читал только `PILOT_ORGANIZATION_ID`.

Полноценный self-service QR-интерфейс подключения WhatsApp (`apps/widget`) — крупная отдельная работа и остаётся Фазой 2; сейчас закрываем дешёвую часть, чтобы можно было заводить повторных клиентов вручную скриптами.

---

## Решение

**Заводим нового клиента тремя идемпотентными скриптами, всё из параметров, ничего про конкретного клиента в коде.**

1. **Лист дебиторки — per-org в БД.** Новое поле `Organization.debtSheetFileId` (nullable). Воркер резолвит источник через `getDebtSourceForOrg(organizationId)`: `file_id` из БД по организации, креды сервисного аккаунта — общие из env (один аккаунт добавляется редактором в таблицу каждого клиента). Фолбэк на env `DEBT_SHEET_FILE_ID` оставлен как мост, чтобы прод-пилот не упал до пересева; `seed.mjs` теперь пишет `debtSheetFileId` в БД, делая её source of truth.
2. **`onboard-org.mjs`** — заводит организацию + дефолтные настройки + расписания из env (`ORG_ID`, `ORG_NAME`, `ORG_DEBT_SHEET_FILE_ID`, `ORG_TELEGRAM_CHAT_ID`, `ORG_OWNER_TELEGRAM_USER_ID`). Без менеджеров и номеров.
3. **`onboard-managers.mjs`** — состав МОП клиента из JSON-файла вне репозитория (`MANAGERS_FILE`): `Manager` + `ManagerSheetAlias`. Замена захардкоженного `MANAGERS` из `seed.mjs` для не-пилотных орг.
4. **`onboard-channels.mjs`** — теперь принимает generic `ORG_ID` (фолбэк на `PILOT_ORGANIZATION_ID`).
5. **Общий модуль `default-settings.mjs`** — `DEFAULT_SETTINGS` + `seedScheduleJobs` разделяются между `seed.mjs` и `onboard-org.mjs`, чтобы стартовые настройки клиентов не расходились.

Порядок онбординга: `onboard:org` → `onboard:managers` → `onboard` (номера).

---

## Последствия

- **Второй клиент заводится без правок кода** — тремя командами + два файла вне репозитория (роль-специфичный `MANAGERS_FILE`, секретный `WA_CHANNELS_FILE`).
- **`seed.mjs` остаётся сидом пилота** (захардкоженный состав Медета) — его не трогаем, прод не ломаем.
- **Техдолг Битрикса:** `Organization.bitrixPortalId` всё ещё обязательное `@unique`-поле (наследие до ADR 0009). `onboard-org.mjs` генерит синтетическое `${orgId}.noop.local`. Чистка поля — отдельным ADR/миграцией, не здесь.
- **Не закрыто (осознанно):** self-service QR-подключение WhatsApp в `apps/widget` — Фаза 2. Пока номера клиента подключаем скриптом при продаже.
- **CLAUDE.md** правится в разделе «текущая стадия» (пункт про Фазу-2-онбординг) — скриптовый онбординг существует, QR-интерфейс остаётся Фазой 2.

---

## Отклонённые альтернативы

- **Сразу строить QR-страницу онбординга.** Крупная работа без подтверждённого спроса Фазы 2; ручное подключение номеров скриптом закрывает потребность на первых клиентах (принцип «не обобщаем под несуществующего клиента»).
- **Оставить `file_id` в env, добавляя per-клиент env-файлы.** Ломается на одной машине с несколькими клиентами и не даёт БД как source of truth. Поле в `Organization` — естественное место (лист привязан к организации).
- **Генерализовать `seed.mjs` в один универсальный скрипт.** Смешало бы пилот-специфичный состав с параметризованным онбордингом; проще держать пилотный seed и отдельные onboard-скрипты.
