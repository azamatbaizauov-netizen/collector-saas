# ADR 0007: План дня по обещанной дате оплаты (колонка G)

**Дата:** 2026-06-21
**Статус:** Принято (уточняет ADR 0004 и критерий покрытия из ADR 0006)

---

## Контекст

План «кому написать сегодня» (ADR 0004) изначально отбирал должников по **дням без
оплаты** (колонка I листа, считается от последней оплаты — колонка H), с порогом
`MANAGER_PLAN_MAX_DAYS = 100`: ≤100 дней — в план менеджера, >100 — в эскалацию владельцу.

На практике этот показатель обманчив. Цикл оптового клиента: получил товар → платит
через оговорённый срок (неделя, две, месяц). «Дни без оплаты» тикают от последней
оплаты независимо от договорённости. Пример: клиент получил товар, по колонке H уже
20 дней без оплаты, но в колонке G (обещанная дата) стоит срок через месяц — он ещё
не просрочил. Слать менеджеру «напомни сегодня» рано: это портит доверительные
отношения и даёт ложный сигнал. И наоборот, «100 дней» — не наш бизнес-показатель.

`promisedDate` (колонка G) уже парсится в `NormalizedDebtRow` и протаскивается в
`DailyTask`, но раньше не влиял на отбор.

## Решение

Главный критерий плана менеджера — **обещанная дата оплаты (колонка G)**, а не дни
без оплаты.

- В план менеджера должник попадает, только если `debt > 0` И обещанная дата
  **наступила или прошла** (`promisedDate <= сегодня`, сравнение по календарной дате).
- **Срок в будущем** → не шлём. **Срок не указан (пустая G)** → не шлём (без срока не
  «напоминаем сегодня»).
- **Эскалация** переезжает с «дни без оплаты > 100» на **«просрочка по сроку G > 30
  дней»** (`ESCALATION_OVERDUE_DAYS = 30`, дефолт, настраиваемый параметром
  `buildDailyPlan` / на организацию через `OrganizationSettings`). Просрочка > 30 дн →
  должник выпадает из плана менеджера и уходит владельцу блоком эскалации с именем МОПа.
- **Сводка владельцу (MORNING_DIGEST)** — его собственные клиенты (строки «Медет») —
  **все с долгом > 0, без фильтра по сроку**: владелец видит всех своих.
- Сортировка плана и эскалации — по просрочке (больше дней просрочки по G — выше), при
  равенстве — больше долг. Дни без оплаты больше не управляют планом.

Реализация — чистая функция `buildDailyPlan(rows, today, escalationOverdueDays)` в
`packages/rules/src/daily-plan.ts`; просрочка считается `daysOverduePromised(G, today)`
по календарным датам в UTC. Задачи идут 09:30/14:00/17:00 по `TZ=Asia/Almaty` (ADR 0006)
≈ та же дата в UTC, поэтому сравнения по UTC-дате для пилота достаточно.

## Последствия

- `DAILY_PLAN` и счётчик покрытия (`DAILY_OVERDUE_CHECK`/`FINAL_COVERAGE`) строятся из
  одного `plan.managers` — фильтр по сроку применяется к обоим автоматически. Уточняет
  последнюю строку ADR 0006: в покрытие идут должники из плана (долг > 0, срок G
  наступил, просрочка ≤ 30 дней), а не «≤ `MANAGER_PLAN_MAX_DAYS`».
- Тексты Telegram: вместо «N дн» (дни без оплаты) — «срок сегодня» / «просрочка N дн».
  Заголовок эскалации — «⚠️ Эскалация (просрочка >30 дней)».
- Качество плана теперь зависит от заполненности колонки G в листе. Должники без срока
  выпадают из плана менеджера (но остаются у владельца, если это его клиенты) — это
  осознанный компромисс: лучше не дёргать менеджера без основания, чем слать по дням.
- `daysWithoutPayment` остаётся в `NormalizedDebtRow` (информативно, для будущего
  рейтинга/тона), но планом не управляет.
