# ADR 0008: События долга из WhatsApp-групп (чеки) через vision-разбор

**Дата:** 2026-06-27
**Статус:** Принято

---

## Контекст

У пилотного клиента менеджеры используют не только личные чаты с должниками, но и **рабочие WhatsApp-группы**. В этих группах фиксируют события по долгу:

- **«Отдал долг»** — должник оплатил; в группу кидают **чек оплаты картинкой** (скриншот банковского перевода / квитанции).
- **«Взял долг»** — клиенту отгрузили товар в долг; в группу пишут факт новой задолженности.

То есть группа — это **оперативная лента движения долга**, параллельная Google-таблице дебиторки (ADR 0003). Сейчас мы её не видим: `inbound-message` отбрасывает все групповые сообщения (`isGroup` → skip), потому что `chatId` группы — это id группы, а не номер контрагента, и для счётчика покрытия (ADR 0006) группы были не нужны.

Три факта, определяющие решение:

1. **Чеки приходят картинками** (`imageMessage`), а не текстом. Наш текстовый `parse-reply` (Haiku) их не разбирает — нужен **vision**.
2. **Группа не привязана к должнику.** В личном чате `chatId` = номер контрагента и касание само клеится к строке дебиторки. В группе номера контрагента нет: о каком должнике речь — видно только из содержимого (имя плательщика на чеке, сумма, текст сообщения).
3. **Источник долга — Google Sheets, не Битрикс (ADR 0003), и мы его читаем, а не пишем.** Долг правит владелец/менеджер в листе. Значит автоматически менять долг по чеку из группы нельзя — это нарушило бы и «не дублируем источник», и принцип 6 CLAUDE.md (человек в цикле для критичных решений).

---

## Решение

**Отдельный пайплайн «событий долга из групп», независимый от счётчика касаний. Vision-разбор чека → карточка владельцу в Telegram на подтверждение. Никакой автоматической записи долга.**

### Поток

1. **Приём.** `inbound-message` для групповых сообщений идёт в отдельную ветку (НЕ пишет `WhatsAppTouch` — иначе исказит покрытие ADR 0006). Берём только осмысленные типы: `imageMessage` (чек) и текст с явным событием долга.
2. **Скачивание медиа.** У `imageMessage` Green API отдаёт `downloadUrl` в `fileMessageData` — добавляем его в `InboundMessage`/`InboundMessageJob` (сейчас не достаём).
3. **Vision-разбор** (новый сценарий `packages/ai/receipts/parse-receipt.ts`, Haiku 4.5 — vision-capable, парсинг): из картинки чека вытаскиваем `сумму, валюту, дату, плательщика, получателя, банк`. Ответ валидируем через Zod, вызов логируем в `AiAction` (как все вызовы Claude).
4. **Классификация события:** `PAYMENT` («отдал долг») или `NEW_DEBT` («взял долг»). Для чека по умолчанию `PAYMENT`; текстовые события классифицируем по смыслу.
5. **Привязка к должнику — полуавтомат.** Кандидаты ищем по имени плательщика и по сумме среди строк дебиторки. Привязку **не считаем достоверной** — окончательно её делает человек.
6. **Человек в цикле.** Бот постит владельцу карточку: «Чек на $X от "Альфа", дата DD.MM — зачесть должнику?» с кнопками-кандидатами + «не зачитывать». Долг в листе **не меняем сами**: владелец, подтвердив, правит лист (или отмечает, что учёл). Мы фиксируем у себя только факт события и решение — для аудита.

### Что появляется/меняется в коде

- `packages/messaging`: парсер достаёт `downloadUrl` групповых картинок; `InboundMessage` несёт медиа-ссылку и `senderPhone` (кто в группе запостил — для аудита).
- `apps/worker/jobs/inbound-message`: ветка для групп → постановка задачи `receipt`/`debt-event`, а не `WhatsAppTouch`.
- `packages/ai/receipts/`: vision-промпт `v1.md` (фронт-маттер: модель, температура, max_tokens) + Zod-схема + snapshot-тесты на 5–10 эталонных чеках.
- `apps/bot`: карточка чека владельцу с кнопками привязки/отклонения.
- `packages/db`: лёгкая таблица события долга (`organizationId`, тип `PAYMENT`/`NEW_DEBT`, сумма BigInt + валюта, дата, плательщик-текст, `sourceGroupChatId`, `senderPhone`, `greenApiMessageId`, статус подтверждения, привязанный телефон должника `?`). Идемпотентность по `(organizationId, greenApiMessageId)`. Это не копия долга, а журнал событий + решений (буфер под задачу, как `WhatsAppTouch` в ADR 0006).

## Последствия

- **Группы не идут в счётчик покрытия.** События долга — отдельный канал; `WhatsAppTouch` и метрика «кто написал должникам» (ADR 0006) не затрагиваются.
- **Точность привязки — главный риск.** Чек редко содержит имя должника в том виде, как в листе. Поэтому привязка — предложение владельцу, а не факт. MVP сознательно останавливается на «извлекли + показали на подтверждение».
- **Долг не меняем автоматически.** Источник истины — лист (ADR 0003), правит его человек. Мы держим журнал событий для прозрачности и будущей сверки «чек был — в листе отражено?».
- **Стоимость.** Vision-вызовы дороже текстовых — каждый логируется в `AiAction` (база тарификации Фазы 2). Разбираем только картинки из групп, не каждое фото в личных чатах.
- **Суммы** — `BigInt` в минорных единицах + код валюты; тенге/доллар не складываем (как ADR 0003).

---

## Отклонённые альтернативы

- **Снять фильтр групп и писать `WhatsAppTouch` по группам.** Сломало бы счётчик покрытия (касание ≠ контакт с должником) и не решило бы привязку.
- **Автоматически зачитывать оплату по чеку (правка листа/долга).** Нарушает принцип 6 (человек в цикле) и «источник долга правит клиент». Высокий риск ложной привязки → испорченная дебиторка.
- **Парсить чек как текст после OCR-сервиса.** Лишняя зависимость; Claude vision и распознаёт, и структурирует за один вызов, с логированием в `AiAction`.

## Реализация (2026-07-10)

Первый срез — «считал и показал владельцу», БЕЗ хранения (осознанное решение владельца):

- Вебхук-парсер достаёт `downloadUrl`, `mimeType`, `fileName`, `senderPhone` из `fileMessageData`/`senderData` (`packages/messaging`).
- Групповые сообщения больше не отбрасываются: `inbound-message` для групп зовёт `processGroupReceipt` (`apps/worker/jobs/group-receipt.ts`), `WhatsAppTouch` при этом НЕ пишется (счётчик покрытия ADR 0006 не затронут).
- Vision-разбор `packages/ai/receipts/parse-receipt.ts` (Haiku 4.5) — и картинки, и **PDF** (Kaspi-чеки в основном PDF): достаём сумму, валюту, дату, плательщика, получателя, банк. Валюту НЕ конвертируем (Kaspi-чек = тенге).
- Добавлен тип **`EXCHANGE_RATE`** — скрины курса обмена (долги в USD, платят в тенге): курс кладём строкой в `exchangeRate`. Расширение исходного `PAYMENT`/`NEW_DEBT`.
- Разобранный чек сразу уходит владельцу в Telegram (`receipt-card.ts`, без кнопок). `docType=OTHER` (фото товара, болтовня) владельцу не шлём.

**Отход от исходного решения:** журнал `ReceiptEvent` в БД в этом срезе НЕ заводим — по решению владельца ничего не храним, только пересылаем. Дедуп приёма обеспечивает `WebhookProcessed` (apps/api), поэтому один чек обрабатывается один раз и без своей таблицы. Соответственно нет и подтверждения владельцем/`AuditLog` по чекам — карточка чисто информационная.

**Сознательно НЕ сделано:** хранение/журнал событий, автопривязка чека к строке должника, запись в лист, логирование vision-вызова в `AiAction`. Всё это — следующие шаги, если владелец захочет, чтобы система *использовала* чеки (сводка, сверка), а не только показывала.

## Пересмотр (2026-07-10): дневная сводка возвратов вместо пофишинговой пересылки

Владелец уточнил цель: не пересылать каждый чек по отдельности, а раз в день получать **сводку — сколько человек сегодня вернуло долг и на какую сумму**. Причём «сколько человек» — не по числу чеков (один менеджер может кинуть несколько чеков + скрин курса по ОДНОМУ платежу одного клиента), а **по смыслу**: число определяет ИИ-рассуждение над всей дневной лентой группы.

Это разворачивает решение «ничего не храним»: агрегировать за день без буфера нельзя (каждый вебхук — отдельный процесс). Появляется:

- **Дневной буфер `GroupDayMessage`** (`packages/db`): копим за сутки ВСЮ ленту группы — текст и строку разбора чека vision'ом (`receiptLine`), с `businessDay` по TZ Almaty. Идемпотентность — `(organizationId, greenApiMessageId)`. Это временный буфер под сводку, НЕ журнал событий и НЕ копия долга (принцип 1).
- **Захват вместо пересылки** (`apps/worker/jobs/group-message.ts`, заменил `group-receipt.ts`): каждое групповое сообщение пишем в буфер; медиа разбираем vision'ом сразу (спред стоимости), `OTHER` не буферим. Карточку по каждому чеку больше НЕ шлём (`receipt-card.ts` удалён).
- **Кластеризация Sonnet** (`packages/ai/receipts/summarize-day.ts`, Sonnet 4.6 — рассуждение): на вход дневной транскрипт по порядку, на выход `{payers:[{name,amount,currency}]}`. Модель отвечает за «кто за чем стоит», код — за арифметику.
- **Сводка в 18:00 Almaty** (`apps/worker/jobs/payment-digest.ts`, `ScheduleJobType.PAYMENT_DIGEST`, cron `0 18 * * *`): число людей = `payers.length`; сумму код складывает **по каждой валюте отдельно** (тенге и доллары не конвертируем — курс неизвестен, чек Kaspi = KZT). Постим в **общий чат [собственник + МОПы]** (`telegramGroupChatId`) — Медет там же её видит, а менеджеры видят разбивку по себе (в отличие от приватного счётчика покрытия ADR 0006). Информационно — долг в листе не меняем (принцип 1, человек в цикле).
- **Разбивка по менеджерам:** в транскрипт каждую строку префиксуем «Менеджер X:» — сопоставляем отправителя в группе (`senderPhone`) с номером канала (`WhatsAppChannel.phone`, оба нормализуем через `normalizePhone`). Sonnet возвращает `manager` на каждый возврат, код группирует: «Адильбек: N клиентов на сумму», «Айнур: …». Атрибуция (кто отчитался) — модель по префиксу; счёт и суммы — код.

**Разделение ролей (принцип 5, «правила, а не магия»):** ИИ — только кластеризация людей и извлечение сумм; суммирование по валютам и подсчёт — детерминированный код.

**По-прежнему НЕ сделано:** автопривязка к строке должника, запись в лист, `AiAction`-лог vision/Sonnet-вызовов, сверка «чек был — в листе отражено». Буфер можно периодически чистить (старше N дней) — пока не автоматизировано.

## Связанные ADR

- ADR 0003 — источник дебиторки Google Sheets (read-only, правит клиент): почему долг не пишем автоматом.
- ADR 0006 — касания WhatsApp и счётчик покрытия: почему группы в покрытие не идут.
- ADR 0010 — подтверждаемая запись остатка в лист: тот же паттерн «карточка владельцу», но из личного чата по тексту.
