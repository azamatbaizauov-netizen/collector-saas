# ADR 0006: Хранение касаний WhatsApp для счётчика покрытия

**Дата:** 2026-06-20
**Статус:** Принято

---

## Контекст

Механика прозрачности пилота (ADR 0004, схема 1+2+4) требует на cutoff публичный счётчик в общем Telegram-чате: «кто из менеджеров связался с должниками из плана дня, а кто нет». Чтобы его собрать, нужно знать факт: менеджер X сегодня написал контрагенту с номером Y.

Green API уже шлёт нам входящие и **исходящие** (`outgoingMessageReceived` — отправленные с телефона менеджера) сообщения на `POST /webhook/whatsapp`. Но обработчик `inbound-message` их только парсил Claude'ом и выбрасывал: персистенция была заблокирована (`TODO`) на резолве `contactId` Битрикса.

Ключевое наблюдение: для счётчика покрытия Битрикс не нужен. План дня (`@repo/rules` → `DailyTask`) уже несёт номер телефона должника в E.164. Касание из вебхука тоже несёт номер контрагента (`chatId` без суффикса). Значит, матчинг «план ↔ касание» делается **по номеру телефона**, а не по `contactId`.

## Решение

Вводим отдельную лёгкую таблицу **`WhatsAppTouch`** (`packages/db/prisma/schema.prisma`):

- `organizationId`, `managerId?` (резолвится из `WhatsAppChannel` по `instanceId`), `phone` (E.164 контрагента), `direction` (`INCOMING`/`OUTGOING`), `greenApiMessageId`, `touchedAt`.
- `@@unique([organizationId, greenApiMessageId])` — идемпотентность на уровне записи (поверх дедупа `WebhookProcessed`).
- Индексы по `(organizationId, touchedAt)` и `(organizationId, managerId, touchedAt)` — выборка касаний за день.

Почему отдельная таблица, а не `Reminder` с `source=MANAGER`:

- `Reminder.contactId` обязателен и завязан на Битрикс (тон, шаблон, канал) — для счётчика это лишняя связанность и блокер (того самого `contactId` у нас нет).
- `WhatsAppTouch` — технический буфер под конкретную задачу (покрытие), а не вторая копия переписки. Не нарушает «не дублируем Битрикс»: храним только факт касания + номер, без содержания.
- Когда появится резолв `contactId` (Фаза 2 / Битрикс-обмен), `Reminder`/`Promise` строятся поверх, а `WhatsAppTouch` остаётся источником для счётчика.

**Поток:** вебхук → `inbound-message` пишет `WhatsAppTouch` (обе стороны) → AI-парсинг (`parse-reply`) только для входящих. Счётчик грузит план дня + сегодняшние `OUTGOING`-касания, по каждому менеджеру считает «написал N из M» по совпадению номеров и постит в общий чат.

**Каденция в общий чат (время Алматы, `TZ=Asia/Almaty`):**

- **09:30** `DAILY_PLAN` — план дня: кому сегодня написать.
- **14:00** `DAILY_OVERDUE_CHECK` (`stage='reminder'`) — напоминание-счётчик: кто сколько написал, кто ещё нет.
- **17:00** `FINAL_COVERAGE` (`stage='final'`) — итог дня: с кем так и не связались («такие-то сегодня не вышли на связь»).

14:00 и 17:00 — один обработчик `processOverdueCheck` с разным `stage` (разные тексты). Два разных `ScheduleJobType`, т.к. `ScheduleJob` уникален по `(organizationId, jobType)` — одна строка = один cron.

## Последствия

- Матчинг по номеру: оба конца канонизируем через `normalizePhone` (`+7…`/`+996…`), чтобы `chatId` из Green API и телефон из Sheet сходились.
- Касания без распознанного менеджера (`managerId = null`) сохраняются, но в счётчик не попадают — не теряем данные для разбора.
- Счётчик показывает только должников из плана (долг > 0, ≤ `MANAGER_PLAN_MAX_DAYS`); переписка менеджера с не-должниками в покрытие не идёт.
