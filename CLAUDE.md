# CLAUDE.md

Главный контекст-документ проекта. Каждая сессия Claude Code в этом репозитории читает его первым. При изменении архитектуры, стека или фаз — сначала правим этот файл, коммитим, и только потом меняем код. Иначе Claude у двух разработчиков будет работать с разным пониманием системы.

---

## О проекте

Мы — команда из двух разработчиков, делаем **AI-надстройку для управления дебиторской задолженностью и складом для оптовых продавцов**.

**Бизнес-модель в две фазы:**

1. **Фаза 1 (сейчас).** Внедрение для одного пилотного клиента-оптовика. У клиента уже работают: Битрикс24 (CRM), 1С, есть SIP-телефония. У каждого менеджера свой WhatsApp-номер, принадлежащий компании (при увольнении номер остаётся фирме). Деньги получаем за внедрение и сопровождение.

2. **Фаза 2 (через 3–6 месяцев после успешного пилота).** В партнёрстве с этим клиентом упаковываем решение в SaaS-приложение для **маркетплейса Битрикс24**, продаём другим оптовикам по подписке.

**Главное архитектурное решение, следующее из этой модели:** мы делаем приложение для Битрикс24, а не отдельную систему. Битрикс — это и хранилище данных по клиентам/сделкам, и рабочее место финменеджера. Мы добавляем туда AI-логику, кастомные поля, виджеты и фоновые автоматизации. Маркетплейс Битрикс24 — основной канал дистрибуции в Фазе 2.

## Два продукта внутри одного приложения

### Модуль «Коллектор» (дебиторка) — приоритет №1
Менеджеры лично ведут переписку с должниками через WhatsApp со своих рабочих номеров (принадлежат компании). Наше приложение работает в **режиме Мониторинга**: CRM читает все переписки менеджеров через WhatsApp-шлюз и помогает им не упустить ни одного должника.

**Как это работает:**
- Каждый менеджерский номер подключён к WhatsApp-шлюзу (Green API) через QR-сканирование.
- Все входящие и исходящие сообщения поступают нам через webhook — менеджер при этом продолжает переписываться в обычном WhatsApp на телефоне.
- Claude Haiku парсит каждое сообщение: фиксирует касание, обещание оплаты, дату, сумму.
- CRM показывает менеджеру список должников, которым нужно написать/позвонить сегодня.
- Собственник получает в Telegram ежедневную сводку: кто сколько должников проработал, у кого обещания не закрыты.

**Почему не автопилот:** клиенты-оптовики работают в доверительных отношениях. Менеджеры созваниваются по видео, показывают новый товар, шлют фото — живое общение важно. Бот-напоминалка разрушает эти отношения. Задача системы — не заменить менеджера, а сделать его работу с дебиторкой структурированной и видимой.

**Общая функциональность:**
- Считает рейтинги клиентов и кредитные лимиты на основе истории платежей.
- Понимает ответы клиентов (Claude парсит «переведу в среду», фиксирует обещание).
- Шлёт собственнику ежедневные сводки в Telegram, включая активность менеджеров по дебиторке.
- Эскалирует сложные случаи финменеджеру через задачи в Битрикс.

**Важное ограничение:** видеозвонки WhatsApp не отслеживаются (end-to-end шифрование). Регламент: менеджер после звонка фиксирует договорённость текстом в том же чате — или через кнопку «Записать итог» в виджете.

### Модуль «Логист» (склад/закуп) — приоритет №2, после Коллектора
Принимает выгрузку отчёта из 1С (Excel), даёт ABC-анализ, флаги ХИТ/НЕЛИКВИД, расчёт замороженных денег, рекомендации по скидкам и дозаказу. Утренняя сводка собственнику.

## Текущая стадия: MVP Коллектора (Фаза 1, неделя 1–6)

**Делаем сейчас:**
- Локальное приложение Битрикс24 для пилотного клиента (НЕ публичное приложение маркетплейса).
- Получение событий из Битрикса через webhooks/события (новый счёт, оплата, изменение сделки).
- Custom-поля в карточке контакта: рейтинг, кредитный лимит, дата последнего касания.
- Подключение номеров менеджеров к Green API (QR-сканирование), приём всех переписок через webhook.
- Парсинг сообщений Claude Haiku (intent, дата, сумма обещания), запись в карточку клиента.
- **Виджет A** в карточке контакта: долг, рейтинг, история WhatsApp-переписки, AI-выжимка последнего разговора.
- **Общий Telegram-чат [собственник + бот + все МОПы]** — основной канал задач менеджерам (см. ADR 0004): бот адресно постит «кому напомнить сегодня», на cutoff — публичный счётчик «кто связался / кто нет». Заменяет на пилоте Виджет B. Механика прозрачности собственника (1+2+4): общий чат + ежедневная сводка собственнику в личку без запроса подтверждения (MORNING_DIGEST) + публичный счётчик дня.
- Воркер: ежедневный расчёт «кому написать сегодня» для каждого менеджера + проверка покрытия по WhatsApp на cutoff.

**НЕ ДЕЛАЕМ СЕЙЧАС** (если соблазнило — возвращаемся к этому файлу):
- **Автоматические сообщения должникам от системы** — система не пишет сама, только менеджеры. Автопилот перенесён в Фазу 2 или отдельный продукт.
- Публикацию в маркетплейсе Битрикс24 — после стабилизации на пилоте.
- Multi-tenant UI (онбординг новых клиентов, подключение их Битрикса по OAuth) — Фаза 2.
- **Виджет B (дашборд менеджера в Битриксе)** — перенесён в Фазу 2 (см. ADR 0004). На пилоте задачи менеджерам идут через общий Telegram-чат, т.к. в Битриксе пока нет данных по долгам и менеджеры в CRM не заходят.
- Биллинг и тарифы — Фаза 2.
- Модуль «Логист» — после успешного запуска Коллектора.
- AI-обзвоны через SIP — отдельный продукт, не раньше Фазы 2+.
- ML-прогнозы вероятности просрочки — нужны данные минимум за 6 месяцев.
- Любые интеграции, не нужные пилотному клиенту (AmoCRM, МойСклад напрямую и т.д.).

## Архитектурные принципы

Эти принципы — каркас, который позволит из приложения для одного клиента вырастить SaaS без переписывания. Не нарушаются без обсуждения вдвоём и обновления этого файла.

1. **Битрикс — source of truth для бизнес-данных.** Клиенты, сделки, счета, переписка — живут в Битриксе. Мы НЕ копируем их к себе в БД. У нас в БД только то, чего в Битриксе нет: рейтинги (вычисляемые), логи AI-действий, очереди задач, конфигурация по организации.

2. **Multi-tenant с первого дня в БД.** Все наши таблицы с `organization_id`. На пилоте `organization_id` всегда один, но архитектурно второй клиент добавляется без миграций. Изоляция через приложение + row-level security в Postgres.

3. **Адаптеры для внешних систем.** Битрикс, WhatsApp-шлюз, 1С — за интерфейсами (`CrmAdapter`, `MessagingAdapter`). Сейчас реализуем `GreenApiAdapter`. Завтра появится клиент на Wazzup или AMO — пишем новый адаптер, не переписываем половину системы.

4. **Все бизнес-правила — в конфигурации, не в коде.** Пороги рейтингов, тоны напоминаний, шаблоны сообщений, расписания — таблица `OrganizationSettings`. Первый клиент крутит свои значения, десятый — свои.

5. **Правила, а не магия.** Рейтинги, ABC, лестница лимитов — детерминированный код. LLM используем только там, где нужна работа с естественным языком (генерация текста, разбор ответа клиента, объяснения).

6. **Человек в цикле для критичных решений.** Изменение кредитного лимита, перенос даты платежа, эскалация в STOP-список — только с подтверждением финменеджера или собственника одной кнопкой. Никаких автоматических списаний и блокировок.

7. **Локальное приложение → Размещаемое.** На пилоте делаем Локальное приложение Битрикс24 (proще, без модерации, доступ через постоянный токен). Архитектурно сразу учитываем требования Размещаемого приложения: OAuth, корректные scope'ы, публичный HTTPS, iframe-совместимый UI. Переход в маркетплейс — это конфигурация и сертификация, не переписывание.

8. **Русский — первый класс.** UI приложения, тексты от ИИ, шаблоны сообщений — на русском. Английский только в коде, именах файлов, коммитах.

9. **Денег в `float` не храним.** Все суммы — `bigint` в минорных единицах (тиыны/копейки) с указанием валюты.

## Технологический стек

- **Backend:** Node.js 20 + TypeScript (strict) + Fastify
- **Frontend (виджеты и страницы в iframe Битрикса):** Next.js 15 + Tailwind + shadcn/ui
- **БД:** PostgreSQL 16 + Prisma ORM
- **Очереди и расписания:** BullMQ + Redis
- **AI:** Anthropic API
  - Sonnet 4.6 — генерация напоминаний, объяснения, рекомендации
  - Haiku 4.5 — классификация и парсинг (intent ответов, извлечение даты/суммы из текста)
- **Битрикс24 интеграция:** прямые HTTPS-запросы к REST API + webhook'и (своя обёртка в `packages/bitrix`, без тяжёлых SDK)
- **WhatsApp:** Green API — шлюз на базе WhatsApp Web (~$12/мес за номер). Каждый менеджерский номер = отдельный экземпляр. Все входящие/исходящие → webhook → наш api-сервер. Интерфейс в коде: `MessagingAdapter` / `GreenApiAdapter` в `packages/messaging/`.
- **Telegram:** grammy (бот в общем чате [собственник + МОПы]: задачи менеджерам, публичный счётчик, подтверждения собственника — см. ADR 0004)
- **Хостинг:** один VPS (Hetzner CX22 или аналог) с Docker Compose; HTTPS через Caddy с автоматическими LE-сертификатами

**Не используем без веской причины:** LangChain/LlamaIndex и подобные AI-фреймворки, GraphQL, микросервисы, Kubernetes, тяжёлые ORM кроме Prisma, любые «универсальные коннекторы» к Битриксу (npm-пакеты от случайных авторов с 50 загрузками — нет).

## Модель данных (только то, что у нас в БД)

**Не дублируем Битрикс.** Клиенты, сделки, счета — берём через API в реальном времени или кешируем коротко (минуты).

**Наши таблицы:**

- `Organization` — клиент сервиса. Хранит: id портала Битрикс, токены, базовые настройки.
- `OrganizationSettings` — JSON с конфигурацией бизнес-правил (пороги рейтингов, тоны, расписания, шаблоны).
- `BitrixUser` — соответствие пользователей Битрикса нашим ролям (если нужно расширить базовые).
- `CustomerRating` — текущий рейтинг клиента + история изменений. `contact_id` (битрикс) + `organization_id`.
- `CreditLimit` — текущий лимит + история изменений.
- `Reminder` — каждое касание по дебиторке: канал, шаблон, текст, время, статус, ответ клиента, intent. Поле `source: 'auto' | 'manager'` — кто инициировал (система или менеджер сам написал).
- `Promise` — обещание оплаты, извлечённое из ответа клиента. Дата, сумма, статус выполнения.
- `AiAction` — лог каждого вызова Claude API: scenario, prompt-версия, input-summary, output, latency, cost.
- `WhatsAppChannel` — конфигурация подключённых номеров: manager_id, instance_id в Green API, статус подключения.
- `ScheduleJob` — расписания обработки (ежедневное, ежечасное и т.д.).
- `AuditLog` — все значимые действия для разборок.

Реальная схема — в `packages/db/prisma/schema.prisma`. Все таблицы с `organization_id`, кроме `Organization` и `AiAction` (там через `organization_id` тоже, но nullable для системных вызовов).

## Бизнес-логика (значения по умолчанию для пилота)

Все эти значения дублируются в `OrganizationSettings` и могут переопределяться на клиента. Цифры — стартовые, Алдияр (партнёр-финансист) согласует с реальной практикой оптового рынка КЗ перед запуском.

### Рейтинг клиента
Пересчёт раз в сутки воркером. Дефолтные правила:
- **Надёжный**: 0 просрочек за 6 мес, ≥5 оплат за период.
- **Нормальный**: 1–2 просрочки ≤7 дней.
- **Риск**: 3+ просрочки или одна >14 дней.
- **Опасный**: просрочка >30 дней или 2+ невыполненных обещания подряд.
- **STOP**: просрочка >60 дней ИЛИ долг > кредитного лимита × 2.

Изменения пишутся в `CustomerRating` и в timeline Битрикса (запись в карточке клиента). Финменеджер может переопределить вручную с обязательным обоснованием.

### Лестница кредитного лимита
Воркер предлагает изменения; **применяется после подтверждения** собственником через кнопку в Telegram или в Битриксе:
- 3+ оплаты подряд вовремя → +20%.
- Новая просрочка → −15%.
- Переход в `Опасный` или `STOP` → лимит = 0.

### План дня менеджеру («кому написать сегодня»)
Главный показатель — **обещанная дата оплаты (колонка G листа)**, а НЕ дни без оплаты (ADR 0007). Дни без оплаты считаются от последней оплаты (колонка H) и обманчивы: клиент мог получить товар неделю назад и договориться заплатить через месяц — по колонке H уже 20 дней, но срок ещё не наступил, требовать долг рано.
- В адресный список менеджера попадает должник с долгом > 0, **только если его обещанная дата (G) наступила или прошла**. Срок в будущем → не шлём. **Срок не указан (пустая G) → не шлём** (без срока не «напоминаем сегодня»).
- Просрочка по сроку G **> 30 дней** — это уже не «напомнить сегодня», а эскалация/разбор: из плана менеджера выпадает, но не теряется — уходит собственнику в `MORNING_DIGEST` отдельным блоком «⚠️ Эскалация (просрочка >30 дней)» с указанием ответственного менеджера (`buildDailyPlan` → `escalationTasks`).
- Порог касается только списков менеджеров. Собственник в `MORNING_DIGEST` видит **всех** своих должников (строки «Медет») с долгом > 0 независимо от срока, плюс блок эскалации по должникам менеджеров с просрочкой >30 дней.
- Дефолт `ESCALATION_OVERDUE_DAYS = 30` в `packages/rules/daily-plan.ts`, переопределяется на организацию через `OrganizationSettings`.
- **Подозрение на тенге-выброс** (`currencySuspect`, ADR 0003: долг > $100k или лимит > $50k — нереально для USD). Такую строку НЕ показываем как $-долг ни менеджеру, ни владельцу: изымаем из планов и эскалации, выносим в `MORNING_DIGEST` отдельным блоком «⚠️ Проверить валюту» (сумма без символа валюты). Это data-quality — чинит владелец/админ в листе, а не задача менеджеру (`buildDailyPlan` → `currencyReview`).

### Тон напоминания
Зависит от дней просрочки:
- 0–3 дня: мягкое.
- 4–10: настойчивое.
- 11–30: жёсткое.
- 30+: финальное + создание задачи финменеджеру в Битриксе.

### ABC и флаги товаров (Логист, на Фазу 1.5)
- **A**: верхние 80% выручки за 90 дней. **B**: следующие 15%. **C**: остальные 5%.
- Флаги: **ХИТ** (A + оборачиваемость <14 дн), **СТАБИЛЬНЫЙ** (A/B + 14–45 дн), **РИСК** (45–90 дн), **НЕЛИКВИД** (>90 дн или 0 продаж 30+ дн).

## Использование Claude API

Все вызовы — через `packages/ai/`. Никаких прямых `fetch` к Anthropic из бизнес-кода.

| Сценарий | Модель | Файл |
|---|---|---|
| Генерация напоминания должнику | Sonnet 4.6 | `packages/ai/reminders/generate.ts` |
| Парсинг ответа клиента | Haiku 4.5 | `packages/ai/conversations/parse-reply.ts` |
| Объяснение тренда товара | Sonnet 4.6 | `packages/ai/logistic/explain-trend.ts` |
| Рекомендация скидки на неликвид | Sonnet 4.6 | `packages/ai/logistic/discount-suggest.ts` |

**Правила:**
- Промпты — в `packages/ai/prompts/<scenario>/v1.md` с фронт-маттером (модель, температура, max_tokens). Версионируем — старые не удаляем.
- Snapshot-тесты с 5–10 эталонными примерами на каждый сценарий.
- Перед мерджем нового промпта — ручная проверка вывода вдвоём.
- На вход модели — минимум данных. Никаких «вот вся карточка клиента, разбирайся».
- Ответы модели валидируем через Zod, прежде чем записать в БД.
- Каждый вызов логируется в `AiAction` с подсчётом стоимости — это база для тарификации в Фазе 2.

## Использование Bitrix24 API

Доступ — только через `packages/bitrix/`. Прямые вызовы из бизнес-кода запрещены.

**Ключевые правила:**
- Используем REST API с правильными scope'ами: `crm`, `imopenlines`, `task`, `user`, `placement`, `event` минимум.
- Все вызовы — с экспоненциальной задержкой при rate-limit (Битрикс отдаёт 503).
- Batch-запросы (`batch` метод) — везде, где возможно. Лимиты Битрикса жёсткие.
- Webhook-обработчики идемпотентны: один и тот же event_id обрабатываем один раз (через табличку `WebhookProcessed`).
- Кастомные поля создаём через `crm.contact.userfield.add` при установке приложения, не вручную. Префикс полей: `UF_AICOL_*` (AI Collector).
- Виджеты регистрируем через `placement.bind`. Placements, которые используем: `CRM_CONTACT_DETAIL_TAB`, `CRM_DEAL_DETAIL_TAB`.
- Записи о действиях AI пишем в timeline сделки/контакта через `crm.timeline.comment.add` — финменеджер видит их в родном интерфейсе Битрикса.

Для пилота — Локальное приложение с одним токеном (`B24_INCOMING_WEBHOOK`). Для маркетплейса (Фаза 2) — полный OAuth flow.

## Использование WhatsApp-шлюза (Green API)

Доступ — только через `packages/messaging/`. Прямые вызовы к Green API из бизнес-кода запрещены.

**Архитектура подключения:**
- У пилотного клиента 5–10 менеджеров, у каждого свой WhatsApp-номер.
- Номера **юридически оформлены на организацию** — при увольнении менеджер оставляет номер компании.
- Каждый номер = отдельный экземпляр (instance) в Green API (~$12/мес).
- Подключение: менеджер один раз сканирует QR в нашем интерфейсе настроек — дальше его переписки поступают нам через webhook.
- Менеджер продолжает пользоваться обычным WhatsApp на телефоне — ничего не меняется в его работе.

**Правила:**
- Входящие и исходящие сообщения всех менеджеров → `POST /webhook/whatsapp` нашего api-сервера → очередь BullMQ → Claude Haiku парсит.
- Обрабатываем **все** чаты каждого менеджера (не только с должниками) — система сама определяет по номеру контакта, есть ли у него дебиторка.
- Каждое касание с должником → запись `Reminder` с `source: 'manager'`.
- Если Claude распознал обещание → запись `Promise`.
- Webhook-обработчики идемпотентны: дедупликация по `message_id` через `WebhookProcessed`.
- Ответственность за содержание переписок — на организации-клиенте (номера их, переписки их).

**Виджеты:**
- **Виджет A** (`CRM_CONTACT_DETAIL_TAB`): показывает историю WhatsApp-переписки с конкретным должником + AI-выжимку (обещания, даты).
- **Виджет B** (дашборд менеджера) — перенесён в Фазу 2 (см. ADR 0004). На пилоте его роль выполняет общий Telegram-чат: план на сегодня + публичный счётчик покрытия. Кнопку «Написать в WA» (`https://wa.me/<phone>`) можно дать прямо в Telegram-сообщении.

## Соглашения о коде

- TypeScript strict. `any` запрещён, используем `unknown` + сужение.
- Имена: файлы `kebab-case.ts`, компоненты `PascalCase.tsx`, переменные `camelCase`, константы `SCREAMING_SNAKE_CASE`.
- Один React-компонент = один файл. >200 строк — разбиваем.
- Бизнес-логика НЕ в React-компонентах и НЕ в HTTP-обработчиках. Только в `packages/rules/` или сервис-функциях.
- Прямой SQL — через Prisma. Raw-запросы только с комментарием `// raw: <причина>`.
- Каждый эндпоинт валидирует вход через Zod.
- Тесты: `vitest` для юнитов, `playwright` для критичных e2e (установка приложения, отправка напоминания, парсинг ответа). Не пишем тесты на тривиальные геттеры.
- Логи: `pino` с структурированными полями (organization_id, contact_id, scenario). Никаких `console.log` в коммитах.

## Как работать с Claude Code

Мы оба новички, поэтому держим дисциплину:

- **Сначала контекст, потом задача.** «Я в `apps/api`, делаю обработчик webhook'а от Битрикса на событие `ONCRMDEALUPDATE`. Нужно проверить, не сменился ли статус сделки на "просрочена".»
- **Одна сессия Claude Code = одна фича.** Не смешиваем модули, контекст замусоривается.
- **Перед серьёзными изменениями — план.** «Сначала опиши план без кода, обсудим, потом реализуем.»
- **Большие рефакторинги — поэтапно.** Не «перепиши модуль», а «шаг 1: вынеси X в сервис, прогони тесты; шаг 2: …».
- **При архитектурной новой идее** — сначала правим этот CLAUDE.md и `docs/adr/`, потом код.
- **Полезное:** `/clear` между задачами, `/compact` если сессия раздулась, `/status` для остатка лимита.

## Анти-паттерны (с пояснениями, почему)

- **Не строим свой UI вместо Битрикса.** Финменеджер работает в Битриксе. Наш UI — это виджеты В Битриксе и страница настроек приложения. Если ловите себя на мыслях «давай нарисуем красивую таблицу дебиторки» — стоп, она уже есть в Битриксе.
- **Не дублируем данные Битрикса у себя.** Кэш минут — да, копия — нет. Иначе на третий месяц получим расхождения и неделю на их разбор.
- **Не идём напрямую в 1С.** Только через стандартный обмен Битрикс↔1С, который у клиента уже работает. Если в обмене чего-то нет — сначала просим клиента дополнить обмен, и только если совсем никак — Excel-выгрузка.
- **Не делаем «умное» там, где нужно правило.** Если фича звучит как «ИИ сам поймёт и решит» — сначала пишем правила, LLM — обёртка.
- **Не автоматизируем критические действия без подтверждения человека.** Списания, блокировки, изменения лимитов — через одобрение.
- **Не добавляем зависимости без обсуждения вдвоём.** Каждый `pnpm add` — это будущая боль.
- **Не «обобщаем» под второго клиента, пока его нет.** Конфигурируемость — да (через `OrganizationSettings`). Абстракции «на все случаи» — нет. Делаем под первого, обобщаем после второго реального запроса.
